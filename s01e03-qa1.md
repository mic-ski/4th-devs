## 1. `const` arrow functions vs `function` declarations — execution order

Both work identically at runtime. The difference is hoisting:

```js
// function declaration — hoisted, can be called before it's defined
greet();
function greet() { console.log("hello"); }  // works fine

// const arrow — NOT hoisted, must be defined before it's called
greet();  // ReferenceError: Cannot access 'greet' before initialization
const greet = () => { console.log("hello"); };
```

**The aha moment:** using `const` arrows enforces the same top-to-bottom execution order that Python and most other languages have by default. Hoisting is a JS-specific quirk. Writing everything as `const` arrows makes the code behave more predictably — define before use, always.

## 2. Even local connections need `await`

"Local" doesn't mean synchronous. The MCP client connects to the server over stdio — even on the same machine, that involves I/O: spawning a process, opening a channel, doing a handshake. Node.js handles all I/O asynchronously regardless of whether it crosses a network.

The distinction is not **local vs remote** — it's **CPU work** (synchronous) vs **I/O or inter-process communication** (always async, always needs `await`).

---

## 3. Without `await` you get a Promise, not the value

When you call an `async` function, it immediately returns a **Promise** — it doesn't wait for the work inside to finish. `await` pauses execution on that line until the Promise resolves, then gives you the actual value.

```js
const mcpClient = createMcpClient(mcpServer);
// mcpClient is a Promise — not a connected client
mcpClient.doSomething();  // TypeError: not a function

const mcpClient = await createMcpClient(mcpServer);
// execution pauses here until the connection is ready
mcpClient.doSomething();  // works fine
```

**Mental model:** a Promise is like a ticket at a restaurant counter — it represents your order, but it's not the food. `await` is you standing at the counter until they call your number and hand you the actual thing. Without `await` you're walking away with just the ticket and then trying to eat it.

The work still happens in the background either way — the difference is whether your next line of code gets the result or the ticket.

## 4. `Object.entries()` and `Object.fromEntries()` are exact opposites

`.map()` only works on arrays — not objects. So the standard pattern for transforming object values is:

```js
const obj = { a: 1, b: 2, c: 3 };

const doubled = Object.fromEntries(
  Object.entries(obj).map(([key, val]) => [key, val * 2])
);
// { a: 2, b: 4, c: 6 }
```

- `Object.entries()` — object → array of `[key, value]` pairs
- `.map()` — transform each pair
- `Object.fromEntries()` — array of pairs → object

**The aha moment:** `.map()` is arrays-only. This three-step pattern is the workaround when you want to transform an object.

## 5. Curly braces mean opposite things depending on which side of `=` they're on

```js
const { tools } = response;        // LEFT side — destructuring, unpacking a property OUT of an object
const response = { tools: [...] };  // RIGHT side — creating a new object
```

Same syntax, opposite operations. The side of the `=` is the only clue.

## 6. Two separate things: schema for the LLM vs handlers for your code

The LLM and your runtime need completely different things from a tool:

- **`tools`** (schema) — sent to the LLM so it knows what tools exist and what arguments they take. Just descriptions, no code.
- **`handlers`** — the actual functions that run when the LLM decides to call a tool. Never seen by the LLM.

The LLM never sees or runs any code. The flow is:

1. You send the schema to the LLM — tool names, descriptions, argument types
2. The LLM decides which tool to call and with what arguments, and returns that decision as JSON
3. Your code reads that decision, looks up the tool in `handlers`, and executes it
4. You send the result back to the LLM so it can form a response

**Mental model:** the LLM is filling out an order form. It says "I want get_weather with city=Tokyo." Your code is the kitchen — it reads the form and actually does the work. The kitchen never appears on the menu.

## 7. The `ops` lookup table pattern — an alternative to `if/else` chains

```js
const ops = {
  add:      () => a + b,
  subtract: () => a - b,
  multiply: () => a * b,
  divide:   () => b === 0 ? { error: "Division by zero" } : a / b
};

const result = ops[operation]?.();
```

Instead of four `if/else` branches, build an object where keys are operation names and values are functions. Look up by key, call with `()`.

**Why it scales better:** adding a fifth operation is one new line. With `if/else` it's another branch. The `?.` before `()` handles the case where `operation` doesn't match any key — returns `undefined` instead of crashing.

**The `()` at the end is what actually executes the function:**
```js
ops[operation]     // retrieves the function — not called yet, nothing runs
ops[operation]?.() // retrieves the function AND calls it immediately
```
Without `()` you'd just hold a reference to the function, like picking up a tool but never using it.

**Dot vs bracket notation — a key distinction:**
```js
ops.operation    // looks for a key literally named "operation" — wrong
ops[operation]   // looks for a key named whatever the VALUE of the variable is — correct
```
Dot notation is for when you know the key name at write time. Bracket notation is for when the key is dynamic — stored in a variable. Same concept as Python's `dict[key]`.

## 8. Shorthand method syntax in objects

JS lets you define functions inside objects without writing `: function`:

```js
// shorthand
const handlers = {
  calculate({ operation, a, b }) { ... }
}

// same thing written in full
const handlers = {
  calculate: function({ operation, a, b }) { ... }
}
```

The key is `calculate`, the value is a function. The shorthand just drops `: function`.

**No Python equivalent.** In Python you'd define the function separately and reference it:
```python
def calculate(args): ...

handlers = {
    "calculate": calculate   # reference to an already-defined function
}
```

In JS, defining functions inline directly inside objects is idiomatic and very common. Arrow functions have no complexity limit for inline use, unlike Python lambdas which are restricted to a single expression.

## 9. Destructuring at the parameter level

You can unpack an object argument right in the function signature:

```js
// receive whole object, extract later
function calculate(args) {
  const operation = args.operation;
  const a = args.a;
}

// extract immediately at the parameter level
function calculate({ operation, a, b }) {
  // operation, a, b are ready to use
}
```

The object still arrives as one argument — destructuring just unpacks it on the way in. Saves three lines and makes the expected shape explicit in the signature.

## 10. Arrow functions — with and without curly braces

```js
// with curly braces — must use explicit return
(tool) => {
  return { name: tool.name };
}

// without curly braces — value is returned implicitly
(tool) => ({ name: tool.name })
```

The extra `()` around the object literal is required — without them, JS reads `{` as the start of a function body, not an object. Wrapping in `()` tells the parser "this is an object, not a block."

## 11. `tools?.length` — one expression, three cases handled

```js
if (tools?.length) { ... }
```

Handles all three failure modes in one go:
- `tools` is `undefined` → `?.` returns `undefined` → falsy → skip
- `tools` is `null` → `?.` returns `undefined` → falsy → skip
- `tools` is an empty array → `length` is `0` → falsy → skip

Only runs when `tools` exists AND has at least one item. The long version would be:
```js
if (tools !== undefined && tools !== null && tools.length > 0)
```

## 12. Zod — define schema and get validation for free

Zod is a general-purpose JS validation library (not LLM-specific). You describe the shape of data you expect, and Zod both validates incoming data at runtime and generates JSON Schema automatically:

```js
// you write this
{ city: z.string().describe("City name") }

// Zod generates this
{
  type: "object",
  properties: {
    city: { type: "string", description: "City name" }
  },
  required: ["city"]
}
```

Without Zod you'd write the JSON Schema by hand and add manual `typeof` checks yourself. Zod collapses both into one line.

## 13. Images in JSON — why base64?

JSON is a text format. Raw image bytes contain values that have no valid text representation and can clash with JSON's own special characters. Base64 converts every byte into a character from a safe 64-character alphabet that JSON handles without ambiguity — at the cost of ~33% size increase.

Alternatives:
- **Multipart form data** — used by browsers when uploading files, sends raw bytes without encoding
- **URL reference** — store the image somewhere, send just the URL in JSON (what most web UIs do)
- **Base64 in JSON** — simpler when you don't have file storage infrastructure; what the Anthropic API uses for image inputs

## 14. `InMemoryTransport` — fake network for testing

The official MCP SDK ships with `InMemoryTransport.createLinkedPair()` — two linked transport objects that simulate a real client/server channel entirely in memory. No network, no stdio, no separate process.

```js
const [clientTransport, serverTransport] = InMemoryTransport.createLinkedPair();
```

One array returned, destructured into two variables. The client and server each get one end and connect through it as if it were real.

**Mental model:** think of it like a pair of walkie-talkies — they come as a linked pair, pre-wired to each other. Whatever one sends, the other receives. `clientTransport` is one walkie-talkie, `serverTransport` is the other. The client and server don't know or care that there's no real radio signal — they just talk through their respective ends.

In production you'd swap `InMemoryTransport` for stdio transport (local process) or HTTP transport (remote server). The tool logic inside the server wouldn't change at all — only the transport layer differs. That's the point of the abstraction.
