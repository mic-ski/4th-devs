  ---
  Improvements over your original proposal

  1. All locations, not just the last one.
  Your proposal said "last coordinates of the person." But the location API
  returns a list of coordinates ‚ÄĒ every place the person was ever spotted. If
  the suspect visited a power plant in the morning but was seen elsewhere later,
   using only "last coordinates" would miss it entirely. calculate_min_distances
   receives the full list and returns the minimum distance across all sightings
  for each plant.

  2. Geocode power plant cities once at startup, not inside the tool.
  The power plants are given as city names ("GrudzińÖdz", "Tczew", etc.), not
  coordinates. To use Haversine, you need lat/lon. Your proposal suggested
  passing a city name to the distance tool on every call. That would mean
  geocoding the same 7 cities 5 times (once per person). Instead, app.js calls a
   geocoder.js function once at startup using Nominatim (a free OpenStreetMap
  geocoding API). The coordinates are stored in memory and passed into the tool
  handlers as a constant. Faster and cleaner.

  ‚Ėé What is Nominatim? It is a free service by OpenStreetMap. You give it a city
   name, it gives back lat/lon coordinates. No API key needed.

  3. Pure JavaScript for Phase 2.
  Already explained above. Math in JS is more reliable than asking an LLM to
  analyze a CSV file.

  4. CSV is appended row by row, not written at the end.
  If the script crashes on person 4 of 5, the first 3 are already saved. You
  won't lose data. This also means you could add a "skip if already processed"
  check later.

  5. The agent loop in Phase 3 asks for confirmation before sending.
  The system prompt for Phase 3 instructs the agent: "Do not call send_answer 
  until the user explicitly confirms." So the agent will show you the full
  answer object and wait for your "yes" before submitting.

  6. The server response is printed.
  The send_answer handler returns the full HTTP response body as a string, which
   the agent will include in its final message. So if the answer is correct, the
   flag ({FLG:...}) will appear in your terminal.

‚ĚĮ Wait, I thought it would be enough to send the schemas once, if they fit in   
the context, why would we resend them every single time...?                     

‚óŹ You're right to question it ‚ÄĒ and your intuition is correct in principle. If 
  the API maintained a persistent context on the server, you could send the
  schemas once and they'd stay there.

  But the API we're using is completely stateless. The server does not remember
  anything between calls. Each call to chat() is an entirely independent HTTP
  request ‚ÄĒ the server has no idea you called it 30 seconds ago. So we have no
  choice but to resend everything every time, including:

  - the system prompt (instructions)
  - the tool schemas (tools)
  - the full conversation history (input)

  It's all just one big JSON blob sent fresh each time.

  ---
  Why is it designed this way?

  Simplicity and scalability. A stateless API means Anthropic/OpenAI can handle
  millions of concurrent users without storing anyone's conversation state on
  their servers. Your app owns the state, the server just processes one request
  and forgets it.

  ---
  Does it waste tokens?

  Yes ‚ÄĒ this is a real cost. The schemas, system prompt, and growing
  conversation are all re-tokenised and re-processed on every round. This is one
   reason why:

  - you want tool descriptions to be concise
  - you want MAX_TOOL_ROUNDS to be a reasonable limit
  - you clear the conversation between persons (our processQuery pattern) ‚ÄĒ
  otherwise the history from person 1 would still be in the payload when
  processing person 5

  ---
  Is there a way around it?

  Some APIs offer context caching ‚ÄĒ you pay to store a chunk of tokens
  server-side (like the system prompt and tool schemas) and reference it by ID
  in future calls. OpenAI calls this prompt caching, Anthropic has it too. It
  reduces cost and latency for the repeated parts. But the underlying model is
  still stateless ‚ÄĒ caching is just an optimisation, not true persistent memory.
