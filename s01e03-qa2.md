## 1. `fetch()` does two things at once

`await fetch(url, options)` both **sends** the full request (headers + body) and **waits** for the server's response headers to arrive. The sending itself doesn't need an `await` ‚Äî it's just the necessary first step. You only wait for what comes back.

```js
const response = await fetch(ENDPOINT, {
  method: "POST",
  headers: { ... },
  body: JSON.stringify(body)   // ‚Üê this is what you SEND
});
```

**Mental model:** `fetch` is named from the client's intent ‚Äî "go get this resource." The sending is just the mechanism to make that happen.

---

## 2. HTTP response arrives in two steps ‚Äî and why

HTTP is designed to stream. Headers arrive first (before the body is even finished generating), so you can make early decisions before committing to reading the full payload.

```js
const response = await fetch(...);      // step 1: headers arrive
const data     = await response.json(); // step 2: body arrives + parsed
```

**Why two awaits?** The first waits for the HTTP layer. The second waits for the body to be fully read and parsed into a JS object.

**The silent bug:** Forgetting the second `await` doesn't crash ‚Äî you just get a pending Promise instead of data. No error, wrong result.

**Mental model:** Like a letter arriving in an envelope. The envelope (headers) lands first ‚Äî you can read the sender and subject before deciding whether to open it.

---

## 3. Request body and response body are completely different things

`body: JSON.stringify(body)` inside the `fetch` options is the **request** body ‚Äî the data you're sending to the server. The **response** body is what comes back, accessed via `await response.json()`. They share the word "body" but are opposite ends of the same transaction.

---

## 4. `response` is a handle, not a plain object

After `await fetch()`, `response` is a `Response` instance ‚Äî a built-in class with properties and methods. Not a plain `{}`.

```js
response.status          // number
response.ok              // boolean
response.headers.get('Content-Type')  // still accessible after .json()
await response.json()    // reads + parses the body into a plain JS object
```

`data` (after `.json()`) is the plain object. `response` is the wrapper. Reading the body doesn't destroy the headers ‚Äî they're on separate parts of the object.

---

## 5. `export` marks names, it doesn't send code

`export` just puts a name on the counter. Importing only gets what's on the counter ‚Äî nothing else in the file is reachable, even if exported functions call private helpers internally.

```js
// ai.js
const extractResponseText = (data) => { ... };  // private ‚Äî stays in the back room

export const extractText = (response) => {
  return extractResponseText(response);          // uses the private helper
};
```

The consumer importing `extractText` has no access to `extractResponseText`. The helper can also live in a different file ‚Äî as long as it's imported at the top of the file that needs it, it works identically.

**The aha moment:** A private helper doesn't become public just because a public function uses it. Like a class method calling a private method ‚Äî the private one doesn't leak.

## 6. Object method shorthand ‚Äî there's no `function` keyword

Inside an object literal, methods use a shorthand where `function` is dropped entirely:

```js
// shorthand (what you'll actually see)
const obj = {
  processQuery(query) { ... },
  async processQuery(query) { ... }   // async just modifies it ‚Äî not what makes it a method
};

// expanded (same thing, more explicit)
const obj = {
  processQuery: async function(query) { ... }
};
```

No `const`, no `function` keyword. The tell that it's a definition (not a call) is the `{ }` body after it. `logQuery(query)` has no body ‚Üí it's a call. `processQuery(query) { ... }` has a body ‚Üí it's a definition.

---

## 7. The object-as-instance pattern ‚Äî a factory instead of a class

A common JS pattern: a factory function returns a plain object with methods, used where Python would use a class instance.

```js
export const createAgent = ({ model, tools, instructions, handlers }) => ({
  async processQuery(query) {
    // ...
  }
});

// Usage
const agent = createAgent({ model, tools, instructions, handlers });
agent.processQuery("what is the weather?");
```

`agent` is a plain object. `processQuery` is its one property ‚Äî a function. The dot is just property access, not class method dispatch.

**Mental model:** The object is just a container for the method. It's a deliberate JS pattern for bundling related behaviour into something that feels like an instance without the overhead of `class`.

**The aha moment:** In Python, `obj.method()` means "call a method defined on this class." In JS, the dot always means the same thing ‚Äî access a property. The fact that the property happens to be a function is what makes it callable. There's no separate concept of "method dispatch."

---

## 8. Everything in JS is (basically) an object

Arrays, functions, classes, Date, RegExp ‚Äî all objects under the hood. The dot always means "access a property." Methods are just function-valued properties.

```js
arr.length    // property holding a number (kept up to date by the engine)
arr.push      // property holding a function
arr.push(4)   // access the property, then call it ‚Äî two steps on one line
```

`length` isn't calculated by a function when you read it ‚Äî the engine keeps it updated as the array changes. `push`, `map`, `filter` are the function-valued properties.

**The aha moment:** JS objects are just bags of key-value pairs, where the values can be anything ‚Äî data or functions. A "method" is just a function stored as a value. Classes are syntactic sugar over the same mechanism.

---

## 9. `return` only exits the nearest enclosing function

```js
function outer() {
  function inner() {
    return "done";  // exits inner only ‚Äî outer keeps running
  }
  inner();
  console.log("still runs");
}
```

In the agent loop, `return text` exits `processQuery` (the nearest function). The for-loop exits as a side effect ‚Äî because the function returned, not because of a `break`. The caller gets the value and carries on normally.

---

## 10. Files are just drawers ‚Äî the work is always in functions

When someone says "ai.js communicates with the LLM," they mean "the `completion` function defined in ai.js does that." Files are just a way to group related functions. The actual execution is always a function call, never a file.

**Separation of concerns example:**
- `ai.js` owns everything about talking to and interpreting the API
- `agent.js` owns the loop logic and tool dispatch
- `agent.js` asks `ai.js` "give me the tool calls" ‚Äî it never touches the raw response shape

This boundary means you could swap out the API provider entirely by only changing `ai.js`.
